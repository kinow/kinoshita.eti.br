It is composed by a set of principles:</p> <ul> <li>Collaborative software development</li> <li>Commercial-friendly standard license</li> <li>Consistently high quality software</li> <li><strong>Respectful, honest, technical-based interaction</strong></li> <li>Faithful implementation of standards</li> <li>Security as a mandatory feature</li> </ul> <p>I have been part of <a href="http://commons.apache.org">Apache Commons</a> for a while, and haven&#8217;t really contributed much yet. Maybe because I had lots of projects related to <a href="http://tupilabs.com">TupiLabs</a> as well as my own wedding this year. But I can assure that <em>respectful, honest, technical-based interaction</em> is quite right (not to discredit the other items, of course).</p> <p>Even though there are critics of the Apache Way (<a href="" title="#1">1</a>, <a href="" title="#2">2</a>, <a href="" title="#3">3</a>), it still stands as an important pillar for the Apache Software Foundation, and its principles help to create stable and production ready software, such as <a href="http://hadoop.apache.org">Apache Hadoop</a>, <a href="http://httpd.apache.org">Apache Httpd</a>, <a href="http://commons.apache.org/lang">Apache Commons Lang</a>, among <a href="http://projects.apache.org">others</a>.</p> <p>There are many reviews and comparisons on Apache software (as well as on other software, like JavaScript libraries, Java Web Frameworks, Ruby Web servers and so on). Sometimes, though, these reviews or comparisons can be biased or not well founded. In cases like this, the developers of the tools may be frustrated, or users can be misled and choose the software based on wrong assertions.</p> <p>I have just returned from honey moon, ready to start writing code again, but first I had to read all the unread messages in my inbox. Some were e-mails from <a href="https://www.apache.org/foundation/mailinglists.html">Apache mailing lists</a>. <a href="http://markmail.org/thread/uoh5m55mh4qjybaw">One of these e-mails</a> had Phil Steitz comments on a post by Daniel Wu. </p> <p>Instead of publishing his performance benchmark results of <a href="http://commons.apache.org/pool">Apache Commons Pool</a>, Daniel posted his code to the commons-dev mailing list. Phil Steitz, one of Apache Commons Pool committers replied with questions and a few points that Daniel could use to enrich his benchmark tests.</p> <p>This kind of behavior happens a lot within Apache (at least in the mailing lists that I follow), and it produces a lot of benefits for different parts. </p> <ul> <li>The person writing a review or comparison can get the programmer opinion before actually publishing anything. </li> <li>The programmer can see how other people were testing his/her code.</li> <li>All other commons-pool committers and maintainers, the mailing list readers, and people that found the mailing list archives will be able to read the conversation history.</li> <li>No misguided benchmark results were published (and lots of wrong decisions were avoided).</li> </ul> <p>I keep loving the Apache Way and the resulting community and code around it.
